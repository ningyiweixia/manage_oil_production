# A5、物料与统计分析统一闭环优化设计

## 1. 背景与目标

本次优化聚焦统计分析、A5 系统集成和物料管理三个模块，以“单井作业任务”为共同业务主线，形成可演示、可追溯、可切换正式外部接口的统一闭环。

当前无法取得真实 A5 和物料平台的接口、字段及测试账号，因此首期采用“标准适配器 + 本地模拟数据 + 配置切换”的过渡方式。系统在没有外部平台时完整运行；后续取得正式接口后，只替换适配器与字段映射，不重构业务表、统计口径或前端页面。

核心验收目标是：

```text
审批通过 -> 派工 -> 物料保障 -> A5 下发/施工/完工
        -> 运行状态与物料状态自动联动
        -> 统计自动更新
        -> 单井详情可追溯到原始记录与操作日志
```

## 2. 范围与非目标

### 2.1 本期范围

- A5 回调验签、状态回传、日报、异常和特殊工序同步。
- 本地 A5 模拟适配器与正式 HTTP 适配器的统一接口。
- A5 同步事件持久化、幂等、失败重试、人工重试和审计。
- 物料需求、审核、计划、出库、配送、到场、使用、退库、补料、异常和闭环。
- 物料状态回写修井运行表。
- 统计总览、分类下钻和单井闭环详情。
- 页面与 Excel/Word 导出复用相同统计查询和指标口径。
- Celery worker/beat 生产部署与同步任务监控。

### 2.2 非目标

- 不猜测真实 A5 或物料平台的最终字段定义。
- 不建设独立数据仓库、流式计算平台或完整事件总线。
- 不引入预测模型、自动派工决策或复杂算法推荐。
- 不替代未来外部物料平台的基础资料维护能力。

## 3. 方案选择

采用“统一闭环”方案：外部数据先通过适配器标准化，再进入持久化同步事件；事件处理服务更新运行表和物料汇总；统计服务只读取数据库中的业务事实和同步事件，Redis 仅作为可重建缓存。

未采用局部增强方案，因为三个模块继续各自维护状态会导致统计口径和单井明细不一致。未采用完整事件驱动方案，因为当前接口尚未确定，引入独立消息总线和统计投影会增加首期交付与运维复杂度。

## 4. 总体架构

```text
A5 本地模拟适配器 ─┐
                    ├─> 标准同步事件 -> 验签/幂等/校验/映射 -> 修井运行表
A5 正式 HTTP 适配器 ┘                         │               │
                                             │               ├─> 完井台账
物料内部台账 -> 状态机/数量校验 -> 物料流转日志 ─┴─> 物料汇总 ─┘
                                             │
                                             └─> 统计服务
                                                  ├─ 综合总览
                                                  ├─ 分类下钻
                                                  ├─ 单井闭环详情
                                                  └─ Excel/Word 导出
```

每个组件保持单一职责：适配器只负责获取和标准化数据，同步服务只负责可靠处理事件，物料服务只负责状态与数量规则，统计服务只负责查询和聚合，前端不复制后端业务计算逻辑。

## 5. A5 集成设计

### 5.1 适配器接口

定义统一的 A5 适配器接口，提供以下能力：

- 获取指定日期的作业日报。
- 获取指定日期的异常记录。
- 获取指定日期的特殊工序记录。
- 查询单个作业单状态。

`LocalA5Adapter` 从现有运行表生成确定性的模拟数据，用于本地演示和自动化测试。`HttpA5Adapter` 使用正式基地址、密钥和超时配置访问外部接口。配置项 `A5_ADAPTER_MODE=local|http` 决定实现；`http` 模式必须同时具备基地址、API Key 和 API Secret。

### 5.2 标准同步事件

新增 `integration_sync_event` 表，核心字段为：

- `source_system`：首期固定为 `a5`，为未来物料接口预留。
- `event_key`：来源系统内唯一的幂等键。
- `event_type`：`workorder_status`、`daily_report`、`anomaly`、`special_process`。
- `operation_no`、`occurred_at`、`received_at`。
- `payload_json`、`payload_hash`。
- `process_status`：`PENDING`、`PROCESSED`、`FAILED`、`PENDING_REVIEW`。
- `attempt_count`、`next_retry_at`、`last_error`、`processed_at`。

优先使用“来源系统 + 外部事件号”形成 `event_key`。外部数据没有事件号时，使用来源、作业单号、事件类型、发生时间和载荷摘要生成稳定键。数据库唯一约束保证重复回调不会重复修改业务状态。

### 5.3 处理流程

1. 回调或定时任务接收原始数据。
2. 验证来源与 HMAC 签名；请求头按大小写不敏感方式读取。
3. 标准化并写入 `integration_sync_event`。
4. 命中既有 `event_key` 时返回幂等成功，不重复执行。
5. 校验作业单、状态、时间和关键字段。
6. 执行合法状态迁移并更新运行表。
7. 记录处理结果、同步批次和错误原因。
8. 完工状态释放承包商运力，并触发完井台账更新。

未知状态、缺少作业单或字段无法映射时标记为 `PENDING_REVIEW`，保留原始载荷但不覆盖运行表。连接超时和暂时性错误标记为 `FAILED`，由 Celery 按指数退避重试；达到阈值后停止自动重试并触发告警。

### 5.4 安全配置

- 生产环境不得使用仓库示例密钥或默认管理员密码。
- `http` 模式缺少 A5 密钥时拒绝启动。
- A5 回调始终验证 HMAC；生产环境不允许降级为“只检查请求头存在”。
- IP 白名单使用可信反向代理提供的客户端地址，不能信任任意客户端提交的首个 `X-Forwarded-For` 值。
- SSO 跳转参数继续使用短期令牌并限制重定向路径。

### 5.5 API

- 保留 `POST /api/v1/a5/callback`，改为持久化和处理标准事件。
- 保留 `POST /api/v1/a5/sync/trigger` 和 `GET /api/v1/a5/sync/status`。
- 新增 `GET /api/v1/a5/sync/events`，按状态、类型、作业单和时间筛选事件。
- 新增 `POST /api/v1/a5/sync/events/{event_id}/retry`，人工重试失败或待处理事件。
- 保留 A5 统计和导出接口，但数据来源切换为数据库事件。

## 6. 物料闭环设计

### 6.1 状态机

保留现有状态值并补充缺失阶段，形成：

```text
PENDING -> APPROVED -> PLANNED -> DELIVERED -> IN_TRANSIT -> ARRIVED -> USED -> CLOSED
   └────────────────────────允许取消的阶段────────────────────────────> CANCELED
```

其中 `DELIVERED` 延续现有数据库含义“已出库”，`IN_TRANSIT` 明确表示配送中，`CLOSED` 表示使用、退库和补料数量已经核对完成。禁止跨阶段跳转、状态倒退和通过通用编辑接口绕过状态机。

### 6.2 数量规则

- `planned_quantity <= quantity + supplement_quantity`。
- `delivered_quantity <= planned_quantity`。
- `arrived_quantity <= delivered_quantity`。
- `used_quantity + returned_quantity <= arrived_quantity`。
- 关闭记录时，使用量、退库量和已确认损耗量之和必须等于到场量。
- 补料创建关联物料记录并保存 `parent_requirement_id`，不能直接改写原始需求数量。

新增 `returned_quantity`、`loss_quantity`、`supplement_quantity`、`parent_requirement_id` 等字段。数量使用现有数值类型以降低迁移风险，本期不统一改造历史字段精度。

### 6.3 流转日志与异常

新增 `material_flow_log` 表，记录物料记录 ID、动作、变更前后状态、变更前后数量快照、操作人、操作时间和备注。

物料异常从单一文本扩展为结构化字段：

- `exception_type`：缺料、配送延迟、数量不符、异常损耗等。
- `exception_status`：`OPEN`、`PROCESSING`、`CLOSED`。
- `exception_responsible_party`、`exception_resolution`。
- `exception_resolved_by_id`、`exception_resolved_at`。

保留现有 `exception_reason` 作为异常说明，兼容历史数据。

### 6.4 运行表联动

根据同一 `operation_sheet_id` 下的物料明细计算运行表物料汇总：

- `NOT_READY`：仍有待审核或未计划需求。
- `PARTIAL_READY`：部分已出库或配送中。
- `READY_AT_SITE`：全部所需物料已到场。
- `EXCEPTION`：存在未关闭异常。
- `CLOSED`：所有物料记录已闭环。

异常优先级最高。汇总结果写入运行表的物料状态字段和进度明细，派工及施工页面可直接展示风险。

### 6.5 API 与前端交互

- 通用 `PUT /materials/{id}` 只允许修改非状态字段。
- 新增 `POST /materials/{id}/transition`，提交目标状态、数量和备注。
- 新增 `POST /materials/{id}/supplements`、`POST /materials/{id}/exceptions`。
- 新增 `PATCH /materials/{id}/exceptions/resolve`。
- 新增 `GET /materials/{id}/timeline` 返回流转轨迹。

前端物料页按“需求台账”和“配送闭环”保留现有入口，增加状态步骤条、数量确认对话框、异常处理抽屉和流转轨迹。每个动作按钮只在合法状态和具备权限时出现；后端仍执行最终校验。

## 7. 统计分析设计

### 7.1 查询与口径

继续使用 `StatisticsAnalysisQuery` 作为统一筛选模型，覆盖起止日期、井号、单位、区块、措施类型、队伍、A5 工序、物料状态和项目状态。页面、下钻、单井详情与导出都复用该查询模型。

日期口径明确为业务发生时间：项目统计使用提报时间，运行统计使用计划或实际作业时间，A5 使用事件发生时间，物料使用创建及各阶段确认时间。响应中返回各数据源的可用状态，单一来源不可用时不伪造数据。

### 7.2 三级统计

一级综合总览至少包含：

- 运行任务数、完工率、平均工期和按期完工率。
- A5 同步成功率、失败事件数、待处理事件数、异常数量和特殊工序数量。
- 物料保障率、到场率、使用率、缺料数、延期配送数和未关闭异常数。
- 项目、运行、A5 和物料趋势及分类分布。

二级分类下钻按指标、单位、时间、措施类型、队伍、状态和异常类型返回明细列表。指标卡和图表点击后携带当前筛选条件和指标编码进入下钻，避免前后端重复计算。

三级单井详情按井号和可选运行单号聚合：项目池、审批轨迹、派工与承包商、A5 同步事件、物料流转、完井记录和操作日志。每个统计来源返回业务表名、记录 ID、事件时间和详情入口。

### 7.3 API

- 保留 `GET /api/v1/reports/statistics-analysis` 作为综合总览。
- 新增 `GET /api/v1/reports/statistics-analysis/drilldown`，参数包含 `metric_code` 和统一筛选条件。
- 新增 `GET /api/v1/reports/statistics-analysis/wells/{well_no}`，可选 `operation_sheet_id`。
- 保留 Excel/Word 导出接口，直接调用综合统计服务，不单独计算指标。

### 7.4 缓存

统计事实以 PostgreSQL 为准。Redis 仅缓存查询结果，缓存键由统一查询模型和数据版本组成。A5 或物料状态变更后递增相关数据版本；Redis 不可用时直接查询数据库，结果口径不变。

### 7.5 前端

统计页面采用“筛选区、核心指标、趋势与分布、异常质量、下钻明细”结构。筛选条件同步到 URL 查询参数；刷新和分享链接后可恢复同一分析视图。图表只展示后端返回的系列，不在浏览器重新聚合。

单井详情使用侧边抽屉或独立详情区域展示六段时间线：项目池、审批、派工、A5、物料、完井。统计导出复用当前 URL 筛选条件。

## 8. 后台任务与部署

`docker-compose.yml` 增加独立 `celery-worker` 和 `celery-beat` 服务，复用后端镜像、环境变量、应用网络和 Redis。worker 执行同步、重试和报表任务，beat 每 30 分钟触发 A5 同步。两个服务均配置健康检查、非 root 用户、日志轮转和重启策略。

本地无 Redis 时，手动同步保持同步执行，便于演示；生产环境不允许回退到进程内缓存承担幂等锁或任务队列。

监控新增：同步事件处理成功率、待处理数、失败数、重试次数、最近成功同步时间、物料未关闭异常数和统计接口耗时。连续失败达到阈值时复用现有告警 Webhook。

## 9. 错误处理与并发

- A5 重复事件返回幂等成功，并返回既有事件 ID。
- A5 非法状态不修改运行表，事件进入待处理状态。
- 物料状态迁移和数量修改在数据库事务中执行，并对目标记录加行锁。
- 物料并发更新采用 `updated_at` 或版本字段检测覆盖冲突。
- 统计无数据返回零值和空集合；非法日期或指标编码返回明确参数错误。
- 导出失败不生成空文件，并保留 trace ID。
- 审计日志写入失败需要记录服务日志；关键业务事件日志不再静默忽略。

## 10. 权限

- A5 事件列表沿用 `a5:read`，同步和重试使用 `a5:sync`。
- 物料查看沿用 `material:read`，状态动作和异常处理沿用 `material:update`，创建补料沿用 `material:create`。
- 统计总览、下钻和单井详情使用 `report:read`，导出使用 `report:export`。
- 单井详情和下钻明细继续应用所属单位、创建人和业务角色数据范围，不能仅依赖菜单或接口权限。

## 11. 测试与验收

所有实现遵循测试驱动开发，每个行为先编写失败测试，再完成最小实现。

### 11.1 后端测试

- A5 请求头大小写不敏感验签、错误签名和生产环境缺少密钥。
- 有事件号和无事件号两种幂等键，重复回调只处理一次。
- A5 合法状态迁移、未知状态、缺少作业单和失败重试。
- 模拟适配器与正式适配器输出相同标准结构。
- 物料全部合法迁移、非法跳转和取消边界。
- 出库、到场、使用、退库、损耗和补料数量约束。
- 物料流转日志、异常处理和运行表汇总。
- 统计统一筛选传递、指标计算、下钻总数一致、单井追溯和导出口径一致。
- 数据权限覆盖列表、详情、下钻和导出。

### 11.2 前端与运行验收

- TypeScript 类型检查和生产构建通过。
- A5 事件表可筛选失败和待处理记录，并可人工重试。
- 物料页面只能执行合法动作，数量对话框显示剩余可操作数量。
- 综合总览点击后进入保留相同筛选条件的下钻结果。
- 单井详情显示完整六段时间线和原始记录入口。
- 完整演示链路通过：审批通过、派工、物料闭环、A5 施工与完工、统计更新和单井追溯。
- 现有 98 项测试和新增测试全部通过，前端生产构建无错误。

## 12. 兼容与迁移

- 保留现有 A5、物料和统计 URL，新增接口以增量方式提供。
- 现有 `DELIVERED` 物料状态继续表示“已出库”，不批量改写历史值。
- 历史 A5 Redis 缓存只用于迁移期回读；新同步数据写入数据库事件表。
- 新增字段提供安全默认值，Alembic 迁移先建表和列，再创建约束与索引。
- 前端在新字段缺失时使用零值或空集合，支持迁移期间后端滚动升级。

## 13. 交付顺序

1. A5 标准事件、验签修复、幂等与适配器。
2. 物料状态机、数量规则、流转日志与运行表联动。
3. 统计总览扩展、分类下钻、单井详情与导出一致性。
4. 前端三模块页面优化。
5. Celery worker/beat 部署、监控和端到端验收。

该顺序保证每一步都能独立测试，并使统计层建立在已经可靠落库的 A5 和物料数据之上。
