# 数据分析模块优化设计

## 目标

在现有统计分析大屏和 `/api/v1/reports/statistics-analysis` 基础上，形成查询口径一致、结果可追溯、导出与页面一致、数据质量可检查、异常可闭环处理的数据分析模块。

## 实施边界

首期覆盖项目池、审批、修井运行、A5、承包商运力、物料和完井数据。不建设预测模型、自动决策派工或独立数据仓库，也不替代外部系统的基础数据维护能力。

## 架构

后端由统一查询模型驱动各业务统计适配器，聚合为稳定的分析响应契约。数据质量服务使用只读规则检查现有业务表；预警规则与处理记录持久化，并通过独立 API 提供查询、处理和关闭能力。前端保持单一驾驶舱入口，筛选条件同时用于页面查询和文件导出，图表通过可追溯链接进入业务明细。

数据流为：

```text
统一筛选条件
  -> 项目/运行/A5/承包商/物料/完井统计适配器
  -> 指标聚合与数据质量检查
  -> 驾驶舱、预警列表、Excel/Word 导出
  -> 井号/任务/A5 工单/物料/完井明细
```

## 统一查询与指标

查询条件包括起止日期、井号、区块、提报单位、措施类型、队伍、A5 工序、物料状态和项目状态。所有子统计必须显式接收适用条件；不适用条件不得静默改变其他统计口径。

核心指标包括项目总数、待审批数、审批通过率、平均审批时长、平均派工时长、作业单数、完工率、按期完工率、A5 异常数、物料到场率、物料使用率和承包商利用率。每项指标必须有稳定字段名、零数据行为和可追溯来源。

## 数据质量与预警

首期数据质量规则：缺失井号、未关联作业任务、未关联 A5 工单、时间顺序错误、物料使用量大于到场量、完井关键数据缺失、承包商资质临期、外部同步失败。质量问题返回规则编码、严重级别、业务主键、井号、说明和明细入口。

预警持久化包含规则、业务主键、严重级别、状态、责任人、处理意见和时间。首期支持查询、标记处理中和关闭；同一规则与业务主键的活动预警必须幂等。

## API

- `GET /api/v1/reports/statistics-analysis`：统一分析响应。
- `GET /api/v1/reports/statistics-analysis.xlsx`：按当前查询条件导出 Excel。
- `GET /api/v1/reports/statistics-analysis.docx`：按当前查询条件导出 Word。
- `GET /api/v1/analytics/data-quality`：数据质量摘要和问题列表。
- `GET /api/v1/analytics/alerts`：预警列表。
- `PATCH /api/v1/analytics/alerts/{id}`：更新处理状态和意见。

所有接口继续使用 `ApiResponse[T]`、现有 JWT/RBAC 和 `report:read`/`report:export` 权限体系。

## 前端

驾驶舱分为筛选区、KPI、主题图表、数据质量与预警、追溯明细五部分。URL 查询参数是筛选状态的唯一可分享表示；导出直接复用当前参数。图表点击生成明细路由参数，不在前端复制后端统计逻辑。

## 错误处理

无数据返回零值和空集合；非法日期范围返回参数错误；单个外部统计源不可用时返回来源状态和可用的其余统计，不伪造数据；导出失败不生成空文件；预警并发更新使用更新时间或状态条件防止覆盖。

## 测试与验收

后端按 TDD 增加查询传播、指标计算、质量规则、预警幂等、导出一致性和权限契约测试。前端以 TypeScript 构建、契约测试和关键交互测试验证查询恢复、导出参数和下钻。最终必须通过后端完整 unittest、前端生产构建、接口烟雾测试，并人工核对一组固定样本。
