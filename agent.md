# 采油二厂井下作业管理系统 - AI Agent 开发上下文文档

## 一、 项目概述
* [cite_start]**项目名称**：采油二厂井下作业管理系统 [cite: 1]。
* [cite_start]**项目定位**：一个集多方协同、多级审批、跨系统联动、数据分析及自动化文档于一体的综合性井下作业全流程线上管理平台 [cite: 4, 29]。
* [cite_start]**核心价值**：实现闭环的业务管理流程，打通底层系统（A5系统及防偏磨系统）壁垒，提升审批进度透明度，并将海量业务数据转化为直观图表赋能决策 [cite: 4, 24, 25, 26]。

## 二、 核心业务流程
该系统包含四个关键业务阶段，开发时需严格遵循此业务逻辑：
1. [cite_start]**上修项目池提报与多级联审**：基层单位录入待修井号至“上修项目池” [cite: 6][cite_start]。项目池管理单位选井提出申请后，依次流转至地质所/生产指挥中心（产量核实）、工艺所（井况核实） [cite: 6][cite_start]。审批通过的井进入“修井运行表”资源库等待派发 [cite: 6]。
2. [cite_start]**队伍状态提报与智能派工**：承包商每日报备可用队伍数量及状态 [cite: 8][cite_start]。工艺所调度人员根据优先级和队伍状态，从“修井运行表”提取任务进行分配，分配时需引入并发防重机制（Redis 分布式锁） [cite: 8, 88]。
3. [cite_start]**数据抓取与统计分析**：实时/定时抓取外部A5系统中的异常情况、特殊工序等数据，生成图表和定制化报告 [cite: 10, 19]。
4. [cite_start]**工程设计与系统协同**：调用外部系统（防偏磨设计系统、A5系统）数据，通过内置规则引擎校验后，自动生成并渲染 Word/Excel 标准化工程设计方案 [cite: 12, 91, 92, 93]。

## 三、 技术栈与总体架构
[cite_start]本项目采用分层解耦的现代化前后端分离架构 [cite: 31]。AI 在生成代码时必须严格限定在以下技术栈内：
* [cite_start]**后端框架**：`FastAPI` (Python异步Web框架) [cite: 34]。
* [cite_start]**前端框架**：`Vue.js 3` + `Element Plus` [cite: 35]。
* [cite_start]**数据库引擎**：`PostgreSQL 15` (利用 `JSONB` 字段存储非结构化工序/措施数据) [cite: 34, 35]。
* [cite_start]**ORM 与数据校验**：`SQLAlchemy 2.0` (配合 Alembic) + `Pydantic v2` [cite: 35]。
* [cite_start]**异步与缓存中心**：`Celery` (分布式任务队列) + `Redis` (状态缓存/并发悲观锁) [cite: 35]。
* [cite_start]**数据处理与文档生成**：`Pandas` (数据清洗统计) + `python-docx` (Word工程报告组装) [cite: 35]。
* [cite_start]**文件存储**：`MinIO` 对象存储 (兼容S3协议，用于报告模板与产出文档) [cite: 34, 35]。
* [cite_start]**数据可视化**：`ECharts 5` [cite: 35]。

## 四、 关键系统集成规范（重要红线）
系统与核心生产系统“A5系统”对接时，必须遵循以下合规约束：
* [cite_start]**操作跳转**：系统配置专属跳转入口，通过 SSO（单点登录）+ 安全令牌鉴权机制无缝跳转至 A5 系统完成工单下发，无需重复登录 [cite: 99]。
* [cite_start]**状态回调**：A5 系统内工单状态变更（审核、办结等）后，通过 RESTful 接口主动回调本系统的专属 API [cite: 101, 111]。
* [cite_start]**数据抓取底线**：严禁使用 WebScraping 爬虫技术 [cite: 60][cite_start]。必须采用 `Celery` 定时轮询拉取 A5 提供的正式 REST API 或数据库只读中间表（Views） [cite: 60, 103]。
* [cite_start]**接口规范**：全面推行 RESTful 风格（名词复数路由），局部更新强制使用 `PATCH`（如审批流转） [cite: 39, 41, 46][cite_start]。所有响应体由 `Pydantic` 约束（包含 `code`, `msg`, `data`） [cite: 49, 50]。

## 五、 核心数据表与关联模型
[cite_start]数据库模型遵循高内聚、低耦合原则 [cite: 134]，核心表包括：
* [cite_start]**`workover_project_pool` (上修项目池表)**：业务源头，存储待修井基础信息及修井措施（JSONB格式） [cite: 136]。
* [cite_start]**`workover_operation_sheet` (修井运行表)**：核心执行表，关联项目池主键，记录作业起止时间、施工进度、绑定队伍等 [cite: 137]。
* [cite_start]**`contractor_capacity` (承包商运力表)**：管理外包队伍资源，记录每日运力报备与队伍状态 [cite: 138]。
* **`engineering_design_doc` (工程设计文档表)**：关联井号，记录文档版本并存储 MinIO 路径 [cite: 139]。
* [cite_start]**`approval_log` (审批日志表)**：实现全流程审批溯源留痕 [cite: 140]。

## 六、 安全与权限控制
* 采用 `RBAC` 角色权限控制体系（部门、项目、个人三级权限） [cite: 127, 152]。
* [cite_start]身份认证兼容企业 SSO 与 LDAP，搭配 `JWT` 动态令牌机制 [cite: 150]。
* [cite_start]必须实现“前端路由渲染拦截”与“后端接口二次 Token/角色校验”的双重防护机制 [cite: 154]。

## 七、 实时交互与事件驱动要求
系统包含高频的跨角色协同，必须严格落实以下事件驱动机制：
* **WebSocket 全双工通信**：针对“工艺所审核通过”、“调度派工完毕”、“工单被驳回”等核心业务节点，后端必须通过 WebSocket 向目标用户（或前端客户端）推送实时的待办通知与桌面弹窗提醒。
* **异常告警推送**：内置异常检测引擎（如 A5 同步连续失败、并发锁死超时），触发预警时需通过企业微信/内部通的 Webhook 接口实时推送高优告警信息给运维或管理人员。

## 八、 数据库关联关系规范 (ORM 约束)
在使用 SQLAlchemy 构建模型时，必须遵循以下确定的 E-R 映射关系：
* **一对多 / 一对一**：`workover_project_pool` (上修项目池) 与 `workover_operation_sheet` (修井运行表)。一个项目可衍生多次作业。
* **多对一**：`workover_operation_sheet` (修井运行表) 与 `contractor_capacity` (承包商运力表)。一支可用队伍可承接多个不冲突的作业任务。
* **多对多**：基于 RBAC 的权限体系，用户、角色、菜单之间通过中间表实现多对多映射。
* **一对多溯源**：`approval_log` (审批日志) 与各业务表关联，确保单条业务流存在完整的节点审批记录。

## 九、 部署架构与网络规划
AI Agent 在生成基础设施即代码 (IaC)、部署脚本或配置反向代理时，必须符合以下架构规划：
* **容器化交付**：基于 `Docker` 容器化部署，采用 `Uvicorn` 管理并发，`Nginx` 负责反向代理与静态资源分发。
* **三层网络安全域隔离**：
  * **前端接入域 (DMZ区)**：部署 Nginx 反代与负载均衡，统一暴露入口。
  * **应用逻辑域 (APP区)**：部署 FastAPI 后端微服务集群及 Celery 异步引擎，负责与外部系统（A5、防偏磨）进行内网专线通信。
  * **核心数据域 (DB区)**：部署 PostgreSQL (主从热备)、Redis 集群和 MinIO。该区域为最高安全级别，**绝对禁止外部直接访问**。

## 十、 项目实施路线图
开发排期与迭代需匹配 14 周、5 阶段的敏捷交付计划：
* **阶段一 (1-2周)**：蓝图设计、UI原型、基础环境与 CI/CD 搭建。
* **阶段二 (3-6周)**：开发项目池台账、状态机审批流、队伍状态报备（Alpha版）。
* **阶段三 (7-10周)**：对接 A5 系统及防偏磨系统，实现文档自动化渲染与 MinIO 归档。
* **阶段四 (11-13周)**：完成 ECharts 数据大屏开发，进行并发压力测试及用户验收测试 (UAT)。
* **阶段五 (14周+)**：全量发布与生产环境监控。

## 十一、 微观业务规则与参数硬性约束
AI 在编写核心逻辑时，必须严格遵守以下业务参数与规范，严禁自由发挥或常规化硬编码：

* **API 状态码字典**：严禁仅使用基础 HTTP 状态码。业务返回必须包含 `20000`(成功)、`40001`(参数错误)、`40100`(身份失效)、`40300`(越权)、`40900`(并发冲突)、`60001`(外部A5联动失败) 等指定状态码。
* **定时抓取频率**：Celery 定时拉取 A5 系统日报及异常数据的轮询频率必须被硬性设置为 **每 30 分钟** 一次。
* **反硬编码与数据字典**：严禁在代码中写死下拉框选项。“井号资源库”、“修井措施类型库”、“工序标准库”必须作为独立的动态数据维表进行管理并提供前后端动态加载接口。
* **深度安全审计**：对于权限变更、工程文档覆盖等高危操作，后端的只读审计探针不仅要记录操作人、IP、时间，**强制要求记录变更前后的数据快照 (Data Snapshot)**。
* **数据分析锚点**：ECharts 前端渲染与 Pandas 后端分析的核心业务指标必须包含且不限于：作业产量、队伍绩效、异常工单分布、工序耗时。

## 十二、 核心业务状态机与代码模式强制要求
在生成具体的业务逻辑代码时，必须遵循以下状态与架构模式设计：
* [cite_start]**状态枚举常量**：业务流转必须使用严谨的英文状态枚举（Enum）。例如，一键生成“上修申请单”后，系统业务状态必须变更为 `PENDING_GEOLOGY_VERIFY`（待地质所核实） 。后续状态机需依据地质所、工艺所、修井运行表依次严格定义。
* [cite_start]**数据检索 DSL 架构**：在开发“数据统计与分析模块”时，严禁使用裸写 SQL 拼接来处理多条件查询。必须利用 Pandas 构建底层处理引擎，实现将前端传入的多维查询参数自动解析并转换为**标准化的 DSL（Domain Specific Language）查询语句**进行统一筛选与聚合计算 。

## 十三、 生产级资源编排与运维规范
在生成 Docker/K8s 部署脚本或架构配置时，必须严格遵守方案设定的生产级资源配额与高可用指标：
* [cite_start]**微服务高可用节点**：应用网关与前端（推荐 4核/8GB）、核心业务节点（推荐 8核/16GB）均必须配置为 2 台以上的负载均衡集群 。
* [cite_start]**存储与缓存集群**：核心数据库 (PostgreSQL) 需主从热备（16核/32GB）；缓存 (Redis) 与对象存储 (MinIO) 必须采用 3 台及以上节点构建分布式集群 。
* [cite_start]**可观测性监控栈**：运维监控模块必须集成 `Prometheus` (指标采集) + `Grafana` (可视化大屏) ，实时监控 API 响应与数据库读写性能。
* [cite_start]**数据容灾策略**：必须在运维脚本或定时任务中体现：核心业务数据每日增量备份，每周全量备份，且备份文件需异地/异机存储 。

## 十四、 极端安全边界与字段级硬性约束
AI 在编写后端中间件、数据库模型或对接外部系统时，必须无条件遵循以下业务与安全准则：

* **A5 系统双重安全边界**：
  * [cite_start]**跳转安全**：单点跳转 A5 系统时，必须实现前端页面与 A5 系统基于“安全令牌”与“唯一 URL 参数”的双向绑定校验 。
  * [cite_start]**接口白名单**：所有的 API 接口调用除了 HTTPS 和 Token 鉴权外，必须在网关或中间件层面强制配置并校验 **IP 白名单访问限制** 。
* [cite_start]**内置业务角色字典**：在初始化 RBAC 权限体系（或生成 Seed Data）时，必须内置原方案设定的核心业务角色：项目池管理员、业务审核员、承包商操作人员、运维管理员 。
* [cite_start]**运力表单核心字段**：承包商动态报备接口必须强制要求传入“特定施工能力”标签字段（例如：是否具备大修资质） 。
* [cite_start]**文档版本自动控制**：所有经系统自动渲染生成的工程设计文档，在落盘存入 MinIO 前，必须在代码层面强制生成并记录文档版本号（Versioning） 。
* [cite_start]**企业微信推送集成**：异常监控（如指标超标、连续拉取失败）的告警通道必须预留或实现企业微信的 Webhook 推送逻辑 。

## 十五、 微观代码规范与底层优化策略
AI 在进行具体的代码实现时，需注意以下隐藏在方案中的微观技术要求，确保代码达到企业级交付标准：

* **API 标准生命周期**：所有核心业务 API 的处理逻辑必须严格按照“6步法”执行：`请求发起 → 身份认证 → 参数校验(Pydantic) → 事务执行 → 操作留痕(写入审计日志) → 结果返回`。
* **前端路由与 JWT 黑名单**：前端 Vue3 项目必须利用 Vue Router 的路由守卫拦截非法 URL 访问；后端 FastAPI 必须实现基于 Redis 的 JWT 吊销机制（支持主动登出失效）。
* **派工排序逻辑**：针对“智能派工”推荐接口的排序（ORDER BY）逻辑，必须严格基于业务表的 `审批通过时间` 与 `产量优先级` 字段进行联合排序。
* **JSONB 索引优化**：在使用 SQLAlchemy 与 Alembic 创建 PostgreSQL 数据库表时，必须为存储非结构化“修井措施”的 JSONB 字段主动创建 `GIN` 索引，以保障检索性能。
* **Excel 解析引擎**：项目池的“批量导入”功能，严禁使用低效的文件逐行读取，底层必须基于 `Pandas` 库对上传的 Excel 模板进行批量解析与数据清洗。
